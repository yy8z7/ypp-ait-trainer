# 7日刷题通关 - 优化计划 (自动化反馈与分轮刷题)

## 1. 当前状态分析
目前系统会在用户首次答对题目时，弹出“太简单/适中/太难”的自评按钮，依赖用户手动选择来决定该题目后续的复习间隔。
同时，系统每天的刷题任务是一次性将几百道题（例如第一天有 573 题）全部塞入队列，导致用户面临的单次任务量过大，心理压力较重。

## 2. 目标与期望变更
- **自动化反馈判断**：移除需要用户手动点击的自我评价按钮。系统通过记录用户答错的次数来自动判断题目难度。
- **分轮次刷题**：将每天庞大的题目总量拆分为每轮 50 题（或动态指定数量）的小轮次，减轻用户压力，提升碎片化时间的利用率。

## 3. 具体实施步骤

### 3.1 移除用户自评 UI (HTML & CSS)
- 在 `index.html` 中彻底删除 `<div id="feedback-section">` 及其子元素。
- 在 `styles.css` 中删除 `.feedback-section`, `.feedback-buttons`, `.feedback-btn` 等相关样式。

### 3.2 改造核心逻辑 (app.js)
1. **重构每日任务的初始化与持久化**
   - 修改 `state.dailyStats[state.currentDay]` 的数据结构，新增 `targetIds`（今日总任务目标）和 `completedIds`（今日已完成题目）。
   - 保证每日随机混刷（第5、6天）的目标题目在一天内固定，不会因为分轮刷新而改变。

2. **实现“分轮刷题”机制**
   - 修改 `startSession()`：每次启动练习时，计算 `未完成题目 = targetIds - completedIds`。
   - 如果未完成题目大于 0，则从中截取**前 50 道题**作为当前轮次的 `sessionQueue`。
   - 更新 UI：原本“剩余 X 题”的显示逻辑变更为显示**本轮**的剩余题目和预计时间。

3. **实现“自动化难度判定”**
   - 删除原有的 `handleFeedback(level)` 函数。
   - 修改 `checkAnswer()` 函数：
     - **一次答对**：视为“太简单/已掌握”，不再加入复习队列，直接调用 `markQuestionCompleted(id)`。
     - **首次答错**：视为“适中”，系统自动将其加入明日（`currentDay + 1`）的复习队列中。
     - **多次答错**（同一题历史错误次数 > 1）：视为“太难”，系统自动将其加入明日（`currentDay + 1`）和后日（`currentDay + 2`）的复习队列中。

4. **轮次结束与日结逻辑拆分**
   - 新增 `finishRound()`：当 `sessionQueue` 耗尽时调用。如果今日还有剩余未完成的题目，则弹出提示“本轮已完成！今日还剩 X 题，请休息一下再继续”，并返回首页。
   - 将原 `finishSession()` 改名为 `finishDay()`：如果所有轮次均已完成（`remainingIds.length === 0`），则执行原有的打卡、连胜计算、PDF错题集生成等日结逻辑。

## 4. 验收标准
1. 界面上不再出现任何“太简单/适中/太难”的按钮，答对后自动平滑进入下一题。
2. 第一天的 573 题将以每次 50 题的形式分批发给用户，首页进度条将正确反映整体的完成度（已完成/总任务量）。
3. 答错的题目能正确地根据错误次数自动推入后续的复习队列中。
4. 确保修改后“断点续刷（Active Session）”逻辑仍然有效。
